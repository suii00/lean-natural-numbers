## 📋 段階別ベストプラクティス

### Phase 1: プロジェクト設計
```yaml
✅ DO: 現実的目標設定
  - 学習価値を第一優先に設定
  - 完璧性より概念理解を重視
  - 段階的成果を明確化

✅ DO: 複雑性管理計画
  - 困難箇所の事前識別
  - TODO戦略の戦略的配置
  - 認知負荷分散設計

❌ DON'T: 完璧主義設定
  - "エラーゼロ"を初期目標にしない
  - "一発完成"を期待しない
  - "CI通過"を最優先にしない
```

### Phase 2: 技術基盤構築
```yaml
✅ Import戦略:
  Step 1: 最小限探索
    import Mathlib.Algebra.Field.Basic
    → 基本機能確認 → ビルドテスト
  
  Step 2: 段階的拡張
    必要機能発生時のみ追加
    import Mathlib.FieldTheory.Separable
  
  Step 3: 適応的調整
    エラー発生時は代替path探索
    完璧主義回避

❌ 失敗パターン:
  - 大量import一括試行
  - 存在不明APIへの依存
  - エラー時の頑固な固執
```

### Phase 3: 実装アーキテクチャ  
```yaml
✅ 段階的実装パターン:

Level 1 - 基本型定義:
  - 最小限・明確な定義
  - 直感的命名規則
  - derivable instanceのみ使用

Level 2 - 具体例構築:
  - 計算可能な小例から開始
  - 既知構造の活用
  - 視覚的理解促進

Level 3 - 抽象理論統合:
  - TODO戦略で困難分散
  - 概念理解を優先
  - 将来拡張性確保

❌ 失敗パターン:
  - 抽象理論から直接開始
  - 全機能同時実装試行
  - sorry使用の忌避
```

### Phase 4: TODO駆動開発
```yaml
✅ 戦略的TODO配置:

TODO記述テンプレート:
  -- TODO: reason="なぜ困難か"
  --       missing_lemma="必要な補助定理"
  --       priority=high/medium/low
  --       educational_value="学習価値説明"

優先度ガイドライン:
  - High: 核心概念、基本理論
  - Medium: 技術的詳細、補助定理
  - Low: 最適化、エッジケース

❌ 避けるべきパターン:
  - TODO理由不明記
  - 優先度未設定
  - 戦略なしsorry乱用
```
