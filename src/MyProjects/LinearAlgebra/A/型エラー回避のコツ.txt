▌ビルドはグリーンです。お疲れ様です。なぜ一発で型エラーなしに実装できるのですか？

> 嬉しいフィードバック、ありがとうございます。型エラーを避けて一発で通せる理由は、次の習慣と方針にあります。

  - 正しい最小 import を先に決める
      - mathlib4 では API の所在が細分化されているため、まず「どの常識がどこにあるか」を明確化します（例: LinearMap.ker
  は Algebra/Module/Submodule/Ker.lean、LinearMap.range は …/Submodule/Range.lean）。
      - ドット記法が通らない設計点（range の注記）に合わせ、確実に解決する記法（LinearMap.range f）を選びます。
      - ドット記法が通らない設計点（range の注記）に合わせ、確実に解決する記法（LinearMap.range f）を選びます。
  -
  既存の標準補題だけで組む
      - 和に対しては常に map_add、スカラー倍は map_smul/smul_mem、核は mem_ker、像は mem_range の存在表示、といった定番
  だけを使います。
      - 新定義を一切導入せず、等式変形と既存補題の組み合わせに限定することで、型推論の不確実性を排除します。
  -
  記法より素の構文を優先
      - ∘ₗ のような追加記法に依存せず、LinearMap.comp を直接使います。必要 import が増えず、名前解決の失敗も減ります。
  -
  ブルバキ流の「射で読む」設計を徹底
      - 核: v ∈ ker f ↔ f v = 0 に落とし、map_add で即決。
      - 像: y ∈ range f ↔ ∃ v, f v = y を展開し、代表元の和から構成。
      - 直積や射影を使うときは、等式を成分へ射影してから標準補題に当てます（例: congrArg Prod.fst (map_add …)）。
  -
  「曖昧さ回避」のコツ
      - ドット記法（f.ker 等）より LinearMap.ker f を選ぶ。
      - 補題名をそのまま使える形に式を整えて simpa。必要なら ext で成分に還元。
  -
  事前確認の徹底
      - 不明な定義の所在はローカルの mathlib パッケージ（.lake/packages/mathlib）で確認し、存在・ファイル名を照合してか
  ら書きます。
      - これで「名前は合っているが import 不足」という失敗を避けます。

  この組み立て方だと、書く前に使う補題・所在・記法が確定しているので、型エラーに当たりにくくなります。引き続き同じ方針で
  他ファイルも整備できます。
